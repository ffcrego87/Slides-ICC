Deterministic Parsing

The whole idea of grammars and derivation trees is pointless unless there is an efficient way for a given grammar to take a string and find a derivation.

Parsing means getting from a string to the derivation trees and rejecting those strings that are not generated by the grammar.

Let’s look at a common algorithm for parsing in a compiler.

Compilers

A compiler for a language such as C or Java starts by building a derivation tree for the input program.

The derivation tree reveals the structure of the program. This phase of the task is called parsing.

There are two approaches:

  • Top-down approach: You look at the first symbol of the input and try to determine the production. (This is what we did with the conversion from grammar to PDA.) This will work if you are lucky, but this is not normally the case.

  • Bottom-up approach: Start with the input string and work your way back to the variable \$S\$. Note that a context-free grammar is not sufficient to handle a complex task like a full programming language, but the parts that are describable by such mechanisms are usually easy to handle (type-checking and repeats, for example).

Bottom-Up Parsing

In Bottom-Up parsing, we read symbols until we have a group that matches the RHS of a production.

Then replace the group with the LHS of the production.

And repeat. (This produces a rightmost derivation.)

Bottom-Up Parsing: Example

Let us assume the token stream is \$a+b\$ whose parse tree is constructed according to the grammar below.

\$G,\quad S \rightarrow S^{\ast}S \mid S + S \mid S - S \mid a \mid b\$

Shifts & Reductions

Bottom-up parsing consists of steps:

  • Shift the next input terminal onto the stack; or

  • Reduce: replace stacked symbols that are RHS of a production by the LHS variable.

The main question is: should one reduce now, or read the next symbol?

Table-Driven Parser for LR(1) Grammars

We consider a special type of CFG called an LR(1) grammar. An LR(1) grammar is a CFG where looking one symbol ahead is enough to decide whether to shift or to reduce. This has a table that tells us what the next operation should be.

The parser has a current state and a stack. Entries on the stack alternate between states and symbols (terminal or variable).

For each state and possible input symbol, the table specifies one of four possible actions:

  • Shift.

  • Reduce by the specified production.

  • Error. (Indicated by blank entry.)

  • Halt. And accept.

Details of Shifting

  1. Push current state onto stack.

  2. Read next symbol and push onto stack.

  3. Change state as specified in the table.

For example: entry s4 means shift and change to state 4.

Details of Reducing

  1. Pop symbols and intervening states and discard.

  2. Update state: say one has reduced to variable \$X\$. Look at (but do not pop) the stack-top state, say \$q\$. Then look up in the table the new state based on \$q\$ and \$X\$.

  3. Push \$X\$.

Example Reduction

For example: entry R1 means use production 1. Say this is \$C \rightarrow \times B\$ and the stack is:

Construction of an SLR(1) Table

Let’s build a table. Technically, this assumes that the grammar is an SLR(1) grammar, where S stands for simple.

The process has 2 main phases.

  1. determining the states and the shifts, and

  2. determining the reductions

Assume we have the context-free grammar (CFG):

\$S \rightarrow AA\$

\$A \rightarrow aA \mid b\$

Step 1: Find the augmented grammar

The augmented grammar of the given grammar is:

\$S' \rightarrow .S\$

[0th production]

\$S \rightarrow .AA\$

[1st production]

\$A \rightarrow .aA\$

[2nd production]

\$A \rightarrow .b\$

[3rd production]

Step 2: Find LR(0) collections of items

Below is the figure showing the LR(0) collection of items (every possible state progression):

The terminals of this grammar are \$\{a,b\}\$.

The non-terminals of this grammar are \$\{S,A\}\$.

Step 2: LR(0) collections (rules)

  • RULE: If any non-terminal has \$.\$ preceding it, we have to write all its productions and add \$.\$ preceding each of its productions.

  • RULE: From each state to the next state, the \$.\$ shifts one place to the right.

Step 2: LR(0) collections (transitions)

  • \$I_0\$ consists of the augmented grammar.

  • \$I_0\$ goes to \$I_1\$ when the \$.\$ of the 0th production is shifted to the right of \$S\$ (\$S' \rightarrow S.\$). This state is the accepting state and \$S\$ is seen by the compiler.

  • \$I_0\$ goes to \$I_2\$ when the \$.\$ of the 1st production is shifted right (\$S \rightarrow A.A\$). \$A\$ is seen by the compiler.

  • \$I_0\$ goes to \$I_3\$ when the \$.\$ of the 2nd production is shifted right (\$A \rightarrow a.A\$). \$a\$ is seen by the compiler.

  • \$I_0\$ goes to \$I_4\$ when the \$.\$ of the 3rd production is shifted right (\$A \rightarrow b.\$). \$b\$ is seen by the compiler.

Step 2: LR(0) collections (transitions, cont’d)

  • \$I_2\$ goes to \$I_5\$ when the \$.\$ of the 1st production is shifted right (\$S \rightarrow AA.\$). \$A\$ is seen.

  • \$I_2\$ goes to \$I_4\$ when the \$.\$ of the 3rd production is shifted right (\$A \rightarrow b.\$).

  • \$I_2\$ goes to \$I_3\$ when the \$.\$ of the 2nd production is shifted right (\$A \rightarrow a.A\$).

  • \$I_3\$ goes to \$I_4\$ when the \$.\$ of the 3rd production is shifted right (\$A \rightarrow b.\$).

  • \$I_3\$ goes to \$I_6\$ when the \$.\$ of the 2nd production is shifted right (\$A \rightarrow aA.\$). \$A\$ is seen.

  • \$I_3\$ goes to \$I_3\$ when the \$.\$ of the 2nd production is shifted right (\$A \rightarrow a.A\$). \$a\$ is seen.

Step 3: Find FOLLOW of LHS of production

For a variable \$A\$, \$FOLLOW(A)\$ is the set of all possible terminal symbols that can follow \$A\$ in a string derived from \$S\$.

For example here:

  • \$FOLLOW(S) = \{\Delta\}\$

  • \$FOLLOW(A) = \{a,b,\Delta\}\$

Notes on the table

  • \$\Delta\$ is by default a non-terminal that takes the accepting state (end-marker).

  • 0..6 denote \$I_0..I_6\$.

  • \$I_0 \rightarrow I_2\$ on \$A\$, so 2 is in column A, row 0.

  • \$I_0 \rightarrow I_1\$ on \$S\$, so 1 is in column S, row 0.

  • Similarly, 5 is written in column A, row 2; 6 in column A, row 3.

  • \$I_0 \rightarrow I_3\$ on \$a\$, so s3 in column a, row 0.

  • \$I_0 \rightarrow I_4\$ on \$b\$, so s4 in column b, row 0.

Notes on the table, cont’d

  • Similarly, s3 in column a rows 2,3 and s4 in column b rows 2,3.

  • \$I_4\$ is a reduced state (dot at end). It corresponds to production 3 (\$A \rightarrow b.\$). Since \$FOLLOW(A)=\{a,b,\Delta\}\$, write R3 in columns a,b,Δ of row 4.

  • \$I_5\$ is a reduced state for production 1 (\$S \rightarrow AA.\$); \$FOLLOW(S)=\{\Delta\}\$, so write R1 in column Δ, row 5.

  • \$I_6\$ is a reduced state for production 2 (\$A \rightarrow aA.\$); \$FOLLOW(A)=\{a,b,\Delta\}\$, so write R2 in columns a,b,Δ of row 6.

This is too complex to do by hand!

This process sounds rather daunting. Fortunately there is software to do it. The UNIX utility YACC generates LR tables for certain CFGs.
